### General

솔리디티 컴파일러의 동작과 관련된 일반적인 구성입니다.

##### `remappings`

- Type: array of strings (remappings)
- Default: 없음
- Environment: `FOUNDRY_REMAPPINGS` 또는 `DAPP_REMAPPINGS`

`<name>=<target>` 형식의 리매핑 배열입니다.

리매핑은 솔리디티 임포트(import)를 다른 디렉토리로 _다시 매핑(remaps)_합니다. 예를 들어, 다음 리매핑은

```
@openzeppelin/=node_modules/@openzeppelin/openzeppelin-contracts/
```

다음과 같은 임포트를

```solidity
import {Context} from "@openzeppelin/contracts/utils/Context.sol";
```

다음과 같이 변경합니다.

```solidity
import {Context} from "node_modules/@openzeppelin/openzeppelin-contracts/contracts/utils/Context.sol";
```

##### `auto_detect_remappings`

- Type: boolean
- Default: true
- Environment: `FOUNDRY_AUTO_DETECT_REMAPPINGS` 또는 `DAPP_AUTO_DETECT_REMAPPINGS`

활성화된 경우, Foundry는 `libs` 폴더를 스캔하여 리매핑을 자동으로 감지하려고 시도합니다.

`false`로 설정하면 `foundry.toml` 및 `remappings.txt`의 리매핑만 사용됩니다.

##### `allow_paths`

- Type: array of strings (paths)
- Default: 없음
- Environment: `FOUNDRY_ALLOW_PATHS` 또는 `DAPP_ALLOW_PATHS`

solc가 추가 디렉토리에서 소스 파일을 읽을 수 있도록 허용합니다. 이는 주로 `pnpm` 등으로 관리되는 복잡한 작업 공간(workspace)과 관련이 있습니다.

[solc allowed-paths](https://docs.soliditylang.org/en/latest/path-resolution.html#allowed-paths)를 참조하세요.

##### `include_paths`

- Type: array of strings (paths)
- Default: 없음
- Environment: `FOUNDRY_INCLUDE_PATHS` 또는 `DAPP_INCLUDE_PATHS`

기본 임포트 콜백(callback)에 추가 소스 디렉토리를 사용할 수 있게 합니다. 패키지 관리자를 사용하여 설치된 타사 라이브러리와 같이 기본 소스 트리와 관련하여 위치가 고정되지 않은 컨트랙트를 가져오려면 이 옵션을 사용하세요. 여러 번 사용할 수 있습니다. 기본 경로(base path)가 비어 있지 않은 경우에만 사용할 수 있습니다.

[solc path resolution](https://docs.soliditylang.org/en/latest/path-resolution.html)을 참조하세요.

##### `libraries`

- Type: array of strings (libraries)
- Default: 없음
- Environment: `FOUNDRY_LIBRARIES` 또는 `DAPP_LIBRARIES`

다음 형식으로 링크할 라이브러리 배열입니다: `<file>:<lib>:<address>`. 예: `src/MyLibrary.sol:MyLibrary:0xfD88CeE74f7D78697775aBDAE53f9Da1559728E4`.

##### `solc_version`

- Type: string (semver)
- Default: 없음
- Environment: `FOUNDRY_SOLC_VERSION` 또는 `DAPP_SOLC_VERSION`

지정된 경우 자동 감지 시스템(아래 참조)을 재정의하고 프로젝트에 단일 솔리디티 컴파일러 버전을 사용합니다.

엄격한 버전만 지원됩니다(예: `0.8.11`은 유효하지만 `^0.8.0`은 유효하지 않음).

##### `auto_detect_solc`

- Type: boolean
- Default: true
- Environment: `FOUNDRY_AUTO_DETECT_SOLC` 또는 `DAPP_AUTO_DETECT_SOLC`

활성화된 경우, Foundry는 프로젝트를 컴파일하기 위해 적절한 솔리디티 컴파일러 버전을 자동으로 해결하려고 시도합니다.

`solc_version`이 설정된 경우 이 키는 무시됩니다.

##### `offline`

- Type: boolean
- Default: false
- Environment: `FOUNDRY_OFFLINE` 또는 `DAPP_OFFLINE`

활성화된 경우, Foundry는 누락된 solc 버전을 다운로드하려고 시도하지 않습니다.

`offline`과 `auto-detect-solc`가 모두 `true`로 설정된 경우, 필요한 solc 버전이 자동 감지되지만 누락된 버전은 설치되지 _않습니다_.

##### `ignored_warnings_from`

- Type: array of strings (file paths)
- Default: 없음
- Environment: `FOUNDRY_IGNORED_WARNINGS_FROM` 또는 `DAPP_IGNORED_WARNINGS_FROM`

컴파일 과정에서 경고를 무시해야 하는 파일 경로 배열입니다. 이는 알려진 경고를 생성하는 특정 파일 디렉토리가 있고 다른 파일에 영향을 주지 않고 이 경고를 억제하려는 경우에 유용합니다.

배열의 각 항목은 디렉토리 또는 특정 파일에 대한 경로여야 합니다. 예시:

`ignored_warnings_from = ["path/to/warnings/file1.sol", "path/to/warnings/file2.sol"]`

이 구성은 컴파일러가 지정된 경로에서 발생하는 모든 경고를 무시하도록 합니다.

##### `ignored_error_codes`

- Type: array of integers/strings
- Default: 소스의 경우 없음, 테스트의 경우 SPDX 라이선스 식별자 및 컨트랙트 크기
- Environment: `FOUNDRY_IGNORED_ERROR_CODES` 또는 `DAPP_IGNORED_ERROR_CODES`

빌드 중에 무시할 솔리디티 컴파일러 오류 코드(예: 경고) 배열입니다.

유효한 값은 다음과 같습니다:

- `license`: 1878
- `code-size`: 5574
- `func-mutability`: 2018
- `unused-var`: 2072
- `unused-param`: 5667
- `unused-return`: 9302
- `virtual-interfaces`: 5815
- `missing-receive-ether`: 3628
- `shadowing`: 2519
- `same-varname`: 8760
- `unnamed-return`: 6321
- `unreachable`: 5740
- `pragma-solidity`: 3420
- `constructor-visibility`: 2462
- `init-code-size`: 3860
- `transient-storage`: 2394
- `too-many-warnings`: 4591

##### `deny_warnings`

- Type: boolean
- Default: false
- Environment: `FOUNDRY_DENY_WARNINGS` 또는 `DAPP_DENY_WARNINGS`

활성화된 경우, Foundry는 솔리디티 컴파일러 경고를 오류로 처리하여 아티팩트가 디스크에 기록되는 것을 중지합니다.

##### `evm_version`

- Type: string
- Default: prague
- Environment: `FOUNDRY_EVM_VERSION` 또는 `DAPP_EVM_VERSION`

테스트 중에 사용할 EVM 버전입니다. 값은 `london`, `paris`, `shanghai`, `cancun`, `prague` 등과 같은 EVM 하드포크 이름이어야 합니다.

호환성 때문에 특정 EVM 버전에 의존하는 경우 `foundry.toml`에 고정하는 것이 좋습니다:

```toml
evm_version = "paris"
```

##### `revert_strings`

- Type: string
- Default: default
- Environment: `FOUNDRY_REVERT_STRINGS` 또는 `DAPP_REVERT_STRINGS`

가능한 값은 다음과 같습니다:

- `default`: 컴파일러가 생성한 리버트 문자열을 주입하지 않고 사용자가 제공한 문자열을 유지합니다.
- `strip`: 모든 리버트 문자열을 제거하고(가능한 경우, 즉 리터럴이 사용된 경우) 부작용(side-effects)은 유지합니다.
- `debug`: 컴파일러가 생성한 내부 리버트에 대한 문자열을 주입합니다. 현재 ABI 인코더 V1 및 V2에 대해 구현되어 있습니다.
- `verboseDebug`: 사용자가 제공한 리버트 문자열에 추가 정보를 덧붙입니다(아직 구현되지 않음).

##### `extra_output_files`

- Type: array of strings
- Default: 없음
- Environment: N/A

아티팩트 디렉토리의 파일에 기록해야 하는 솔리디티 컴파일러의 추가 출력입니다.

유효한 값은 다음과 같습니다:

- `metadata`: 아티팩트 디렉토리에 `metadata.json` 파일로 기록됨
- `ir`: 아티팩트 디렉토리에 `.ir` 파일로 기록됨
- `irOptimized`: 아티팩트 디렉토리에 `.iropt` 파일로 기록됨
- `ewasm`: 아티팩트 디렉토리에 `.ewasm` 파일로 기록됨
- `evm.assembly`: 아티팩트 디렉토리에 `.asm` 파일로 기록됨

##### `extra_output`

- Type: array of strings
- Default: 아래 참조
- Environment: N/A

컨트랙트의 아티팩트에 포함할 추가 출력입니다.

다음 값들은 Forge에 필요하므로 항상 설정됩니다:

```toml
extra_output = [
  "abi",
  "evm.bytecode",
  "evm.deployedBytecode",
  "evm.methodIdentifiers",
]
```

유효한 값 목록은 [Solidity 문서](https://docs.soliditylang.org/en/latest/using-the-compiler.html#input-description)를 참조하세요(_The available output types are as follows_ 검색).

##### `bytecode_hash`

- Type: string
- Default: ipfs
- Environment: `FOUNDRY_BYTECODE_HASH` 또는 `DAPP_BYTECODE_HASH`

바이트코드에 추가되는 메타데이터 해시의 해시 방법을 결정합니다.

유효한 값은 다음과 같습니다:

- ipfs (기본값)
- bzzr1
- none

##### `sparse_mode`

- Type: boolean
- Default: false
- Environment: `FOUNDRY_SPARSE_MODE` 또는 `DAPP_SPARSE_MODE`

빌드에 대해 [희소 모드(sparse mode)](/forge/reference/build#sparse-mode-experimental)를 활성화합니다.

### Optimizer

솔리디티 최적화기(optimizer)와 관련된 구성입니다.

##### `optimizer`

- Type: boolean
- Default: false
- Environment: `FOUNDRY_OPTIMIZER` 또는 `DAPP_OPTIMIZER`

솔리디티 최적화기 활성화 여부입니다.

> ℹ️ **참고**
> `optimizer_runs`가 0보다 큰 값으로 설정되면 솔리디티 최적화기가 자동으로 활성화됩니다.

##### `optimizer_runs`

- Type: integer
- Default: 200
- Environment: `FOUNDRY_OPTIMIZER_RUNS` or `DAPP_OPTIMIZER_RUNS`

실행 횟수(runs)는 배포된 코드의 각 오퍼코드가 컨트랙트 수명 동안 대략 얼마나 자주 실행될지를 지정합니다. 이는 코드 크기(배포 비용)와 코드 실행 비용(배포 후 비용) 간의 균형 매개변수입니다. `optimizer_runs` 매개변수가 `1`이면 짧지만 비싼 코드를 생성합니다. 반면, 더 큰 `optimizer_runs` 매개변수는 더 길지만 가스 효율적인 코드를 생성합니다. 매개변수의 최대값은 `2**32-1`입니다.

일반적인 오해는 이 매개변수가 최적화기의 반복 횟수를 지정한다는 것입니다. 이것은 사실이 아닙니다: 최적화기는 코드를 개선할 수 있는 한 항상 여러 번 실행됩니다.

##### `via_ir`

- Type: boolean
- Default: false
- Environment: `FOUNDRY_VIA_IR` 또는 `DAPP_VIA_IR`

true로 설정하면 컴파일 파이프라인이 새로운 IR 최적화기를 거치도록 변경됩니다.

##### `use_literal_content`

- Type: boolean
- Default: false

true로 설정하면 컴파일 시 URL 대신 리터럴 콘텐츠만 사용하도록 변경됩니다.

##### `[optimizer_details]`

최적화기 세부 정보 섹션은 솔리디티 최적화기의 동작을 조정하는 데 사용됩니다. 이 섹션에는 여러 구성 가능한 값(각각 부울)이 있습니다:

- `peephole`
- `inliner`
- `jumpdestRemover`
- `orderLiterals`
- `deduplicate`
- `cse`
- `constantOptimizer`
- `yul`

기본값은 솔리디티 [컴파일러 입력 설명](https://docs.soliditylang.org/en/latest/using-the-compiler.html#compiler-input-and-output-json-description)을 참조하세요.

##### `[optimizer_details.yul_details]`

최적화기 세부 정보 섹션의 Yul 세부 정보 하위 섹션은 새로운 IR 최적화기의 동작을 조정하는 데 사용됩니다. 두 가지 구성 값이 있습니다:

- `stack_allocation`: 스택 슬롯을 더 일찍 해제하여 할당을 개선하려고 시도합니다.
- `optimizer_steps`: 적용할 최적화기 단계를 선택합니다.

기본값은 솔리디티 [컴파일러 입력 설명](https://docs.soliditylang.org/en/latest/using-the-compiler.html#compiler-input-and-output-json-description)을 참조하세요.

> ℹ️ **참고**
> `via_ir`을 사용할 때 컴파일러 오류가 발생하면 레거시 `optimizer`를 명시적으로 활성화하고 `optimizer_steps`를 빈 문자열로 남겨 두세요.

### Model checker

솔리디티 모델 검사기(model checker)는 OSX 및 Linux용 솔리디티 컴파일러에서 사용할 수 있는 내장 옵트인(opt-in) 모듈입니다. [솔리디티 컴파일러 문서](https://docs.soliditylang.org/en/latest/smtchecker.html#tutorial)에서 모델 검사기에 대해 자세히 알아보세요.

> ℹ️ **참고**
> 모델 검사기는 Linux에서 `z3` 버전 4.8.8 또는 4.8.14가 필요합니다.

모델 검사기 설정은 구성의 `[model_checker]` 섹션에서 구성됩니다.

모델 검사기는 `forge build`가 호출될 때 실행되며 모든 결과는 경고로 표시됩니다.

모델 검사기를 사용할 때 권장되는 설정은 다음과 같습니다:

```toml
[profile.default.model_checker]
contracts = {'/path/to/project/src/Contract.sol' = ['Contract']}
engine = 'chc'
timeout = 10000
targets = ['assert']
```

검증할 컨트랙트를 설정하는 것은 매우 중요합니다. 그렇지 않으면 사용 가능한 모든 컨트랙트가 검증되어 시간이 오래 걸릴 수 있습니다.

권장 엔진은 `chc`이지만 `bmc`와 `all`(둘 다 실행)도 허용됩니다.

기본 솔버(solver)에 주어진 기본 시간이 충분하지 않을 수 있으므로 적절한 타임아웃(밀리초 단위)을 설정하는 것도 중요합니다.

검증 타겟이 주어지지 않으면 어설션만 확인됩니다.

##### `[model_checker]`

모델 검사기 섹션에서는 다음 키를 사용할 수 있습니다.

###### `model_checker.contracts`

- Type: table
- Default: all
- Environment: N/A

모델 검사기가 분석할 컨트랙트를 지정합니다.

테이블의 키는 소스 파일의 경로이고 값은 확인할 컨트랙트 이름의 배열입니다.

예를 들어:

```toml
[profile.default.model_checker]
contracts = { "src/MyContracts.sol" = ["ContractA", "ContractB"] }
```

###### `model_checker.div_mod_with_slacks`

- Type: boolean
- Default: false
- Environment: N/A

나눗셈 및 모듈로 연산이 인코딩되는 방식을 설정합니다.

자세한 내용은 [솔리디티 문서](https://docs.soliditylang.org/en/latest/smtchecker.html#division-and-modulo-with-slack-variables)를 참조하세요.

###### `model_checker.engine`

- Type: string (아래 참조)
- Default: all
- Environment: N/A

실행할 모델 검사기 엔진을 지정합니다. 유효한 값은 다음과 같습니다:

- `chc`: 제한된 혼 절(constrained horn clauses) 엔진
- `bmc`: 제한된 모델 검사기(bounded model checker) 엔진
- `all`: 두 엔진 모두 실행

엔진에 대한 자세한 내용은 [솔리디티 문서](https://docs.soliditylang.org/en/latest/smtchecker.html#model-checking-engines)를 참조하세요.

###### `model_checker.invariants`

- Type: array of strings
- Default: N/A
- Environment: N/A

모델 검사기 불변성을 설정합니다. 유효한 값은 다음과 같습니다:

- `contract`: 컨트랙트 불변성
- `reentrancy`: 재진입(Reentrancy) 속성

불변성에 대한 자세한 내용은 [솔리디티 문서](https://docs.soliditylang.org/en/latest/smtchecker.html#reported-inferred-inductive-invariants)를 참조하세요.

###### `model_checker.show_unproved`

- Type: boolean
- Default: false
- Environment: N/A

증명되지 않은 모든 타겟을 출력할지 여부입니다.

자세한 내용은 [솔리디티 문서](https://docs.soliditylang.org/en/latest/smtchecker.html#unproved-targets)를 참조하세요.

###### `model_checker.solvers`

- Type: array of strings
- Default: N/A
- Environment: N/A

모델 검사기 솔버를 설정합니다. 유효한 값은 다음과 같습니다:

- `cvc4`
- `eld`: v0.8.18에서 도입됨
- `smtlib2`
- `z3`

자세한 내용은 [솔리디티 문서](https://docs.soliditylang.org/en/latest/smtchecker.html#smt-and-horn-solvers)를 참조하세요.

###### `model_checker.timeout`

- Type: number (milliseconds)
- Default: N/A
- Environment: N/A

기본 모델 검사기 엔진의 타임아웃을 설정합니다(밀리초 단위).

###### `model_checker.targets`

- Type: array of strings
- Default: assert
- Environment: N/A

모델 검사기 타겟을 설정합니다. 유효한 값은 다음과 같습니다:

- `assert`: 어설션
- `underflow`: 산술 언더플로우
- `overflow`: 산술 오버플로우
- `divByZero`: 0으로 나누기
- `constantCondition`: 사소한 조건 및 도달할 수 없는 코드
- `popEmptyArray`: 빈 배열 pop
- `outOfBounds`: 범위를 벗어난 배열/고정 바이트 인덱스 액세스
- `default`: 위의 모든 항목 (참고: Forge의 기본값은 아님)

### Compilation Restrictions

컴파일 제한(Compilation Restrictions)을 사용하면 프로젝트 내의 개별 파일 또는 디렉토리가 컴파일되는 방식을 구성할 수 있습니다.

> 컴파일러 프로필 및 컴파일러 제한에 대한 복잡한 구성은 Uniswap [contracts monorepo](https://github.com/Uniswap/contracts/blob/4d80a9cfed9b920b83a394b168a95319606a0b73/foundry.toml#L27-L72)를 참조하세요.

예를 들어, 아래 구성을 사용하면 `src` 아래의 `Counter.sol` 컨트랙트는 `via-ir` 옵션이 활성화된 상태로 컴파일되는 반면, 다른 모든 컨트랙트(테스트 포함)는 기본 프로필로 컴파일됩니다.

```toml
# via_ir 프로필 추가
additional_compiler_profiles = [ { name = "via-ir", via_ir = true } ]

# via_ir 없이 테스트를 컴파일하고 일부 무거운 컨트랙트만 via_ir로 컴파일하도록 강제
# 참고: 테스트가 vm.deployCode를 사용하여 컨트랙트를 배포하는 경우에만 작동합니다.
compilation_restrictions = [
    { paths = "test/**", via_ir = false },
    { paths = "src/Counter.sol", via_ir = true },
]
```

`compilation_restrictions` 구성을 지정하여 개별 파일 또는 디렉토리가 컴파일되는 방식을 구성할 수 있습니다(제한된 파일을 임포트하는 소스에도 적용됨).
제한 사항 집합이 있으면 이를 충족하기 위한 설정 객체를 구성해야 합니다. 이는 프로필 이름에서 설정 재정의(overrides)로의 매핑인 `additional_compiler_profiles`를 추가하여 수행됩니다.

위의 예제에서는 기본 컴파일 설정을 사용하지만 `via_ir` 옵션을 활성화하는 `via-ir`이라는 단일 프로필을 추가하여, `Counter.sol` 컨트랙트는 `via_ir`로 컴파일하고 다른 모든 컨트랙트(테스트 포함)는 기본 프로필로 컴파일할 수 있게 했습니다.
컴파일러 프로필을 선택할 때, 소스 파일과 모든 임포트의 제한 사항을 충족하는 첫 번째 프로필(기본 프로필부터 시작)이 사용됩니다.

> 동일한 소스 파일이 여러 프로필로 컴파일되면 각 프로필에 대한 아티팩트가 생성됩니다. `Counter`가 기본 프로필과 `v1` 프로필로 컴파일되면 `Counter.json` 및 `Counter.v1.json` 아티팩트가 생성됩니다.

#### 추가 컴파일러 설정:

`[additional_compiler_profiles]`

- Type: array of additional compiler profiles

프로필 이름에서 설정 재정의로의 매핑을 구성합니다.

예를 들어, 컴파일 파이프라인이 새로운 IR 최적화기를 거치도록 변경하는 추가 `via-ir` 컴파일러 프로필은 다음과 같이 정의할 수 있습니다:

```toml
additional_compiler_profiles = [ { name = "via-ir", via_ir = true } ]
```

컴파일러 프로필에는 여러 구성 가능한 값이 있습니다:

- `name`
- `via_ir`
- `evm_version`
- `optimizer`
- `optimizer_runs`
- `bytecode_hash`

##### `additional_compiler_profile.name`

- Type: string

추가 컴파일러 프로필 이름입니다.

##### `additional_compiler_profile.via_ir`

- Type: boolean
- Default: false

추가 컴파일러 프로필 [via_ir](#via_ir) 설정입니다.

##### `additional_compiler_profile.evm_version`

- Type: string
- Default: prague

추가 컴파일러 프로필 [evm_version](#evm_version) 설정입니다.

##### `additional_compiler_profile.optimizer`

- Type: boolean
- Default: false

추가 컴파일러 프로필 [optimizer](#optimizer) 설정입니다.

##### `additional_compiler_profile.optimizer_runs`

- Type: integer
- Default: 200

추가 컴파일러 프로필 [optimizer_runs](#optimizer_runs) 설정입니다.

##### `additional_compiler_profile.bytecode_hash`

- Type: string
- Default: ipfs

추가 컴파일러 프로필 [bytecode_hash](#bytecode_hash) 설정입니다.

#### 컴파일 제한 설정:

#### `[compilation_restrictions]`

- Type: array of compilation restrictions settings

##### `compilation_restriction.paths`

- Type: regex

주어진 경로와 일치하는 파일에만 컴파일 제한을 적용합니다.

예를 들어:

- `src/contracts/Counter.sol` - 특정 컨트랙트에 대한 제한 적용
- `src/contracts/{Counter,CounterFactory}.sol` - 이름으로 여러 컨트랙트에 대한 제한 적용
- `src/v1/**` 또는 `src/v3/**/libraries/**` - 일치하는 소스 경로의 모든 컨트랙트에 대한 제한 적용

컴파일 제한에는 여러 구성 가능한 값이 있습니다:

- `version`
- `via_ir`
- `bytecode_hash`
- `optimizer_runs`
- `min_optimizer_runs`
- `max_optimizer_runs`
- `evm_version`
- `min_evm_version`
- `max_evm_version`

##### `compilation_restriction.version`

- Type: string (semver)

특정 [solc_version](#solc_version)에 대한 컴파일을 제한합니다.

##### `compilation_restriction.via_ir`

- Type: boolean

특정 [via_ir](#via_ir) 설정에 대한 컴파일을 제한합니다.

##### `compilation_restriction.bytecode_hash`

- Type: string

제공된 바이트코드 해시 방법과 일치하는 컴파일을 제한합니다.

##### `compilation_restriction.optimizer_runs`

- Type: integer

특정 [optimizer_runs](#optimizer_runs) 설정에 대한 컴파일을 제한합니다. `min_optimizer_runs` 및 `max_optimizer_runs` 설정과 충돌합니다.

##### `compilation_restriction.min_optimizer_runs`

- Type: integer

[optimizer_runs](#optimizer_runs) 설정의 최소값에 대한 컴파일을 제한합니다. `optimizer_runs` 설정과 충돌하며 `max_optimizer_runs` 설정이 필요합니다.

##### `compilation_restriction.max_optimizer_runs`

- Type: integer

[optimizer_runs](#optimizer_runs) 설정의 최대값에 대한 컴파일을 제한합니다. `optimizer_runs` 설정과 충돌하며 `min_optimizer_runs` 설정이 필요합니다.

##### `compilation_restriction.evm_version`

- Type: string

특정 [evm_version](#evm_version) 설정에 대한 컴파일을 제한합니다. `min_evm_version` 및 `max_evm_version` 설정과 충돌합니다.

##### `compilation_restriction.min_evm_version`

- Type: string

[evm_version](#evm_version) 설정의 최소 버전에 대한 컴파일을 제한합니다. `evm_version` 설정과 충돌하며 `max_evm_version` 설정이 필요합니다.

##### `compilation_restriction.max_evm_version`

- Type: string

[evm_version](#evm_version) 설정의 최대 버전에 대한 컴파일을 제한합니다. `evm_version` 설정과 충돌하며 `min_evm_version` 설정이 필요합니다.
